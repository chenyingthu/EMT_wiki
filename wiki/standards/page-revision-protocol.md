# 页面修订协议 (Page Revision Protocol)

本协议是 EMT Wiki 页面润色、修复和补充的执行规范。它面向并发 Agent 工作：每个 Agent 可以处理不同页面 shard，但必须遵守相同的证据纪律、页面职责和交付要求。

本协议建立在以下文件之上：

- `schema/WIKI.md`：页面类型、目录、frontmatter 和 wikilink 约定。
- `schema/QUALITY.md`：A/B/C/D 质量等级和质量红旗。
- `wiki/standards/terms-and-notation.md`：术语、符号和表达规范。
- `wiki/standards/page-revision-registry.md`：已修订页面的状态、责任和保护登记。
- `plans/wiki-page-revision-agent-workplan.md`：并发执行计划、分片方式和验证门槛。

## 1. 基本原则

- 页面质量不等于篇幅长度；关键是定义清楚、机制可懂、证据可追溯、边界明确。
- 不把单篇论文结论写成领域共识。
- 不把工具平台当作论文贡献。
- 不把开放研究方向写成工具、模型或方法已经具备的能力。
- 不新增无来源的数字结论。
- 不破坏 frontmatter、来源路径、deep-review/deep-enrich 标记和已有可验证证据。

## 2. 页面职责

| 页面类型 | 核心职责 | 常见错误 |
|----------|----------|----------|
| source | 说明一篇论文做了什么、如何验证、支撑哪些页面 | 把单篇结果外推成领域结论 |
| topic | 综合一个研究方向的定义、分支、趋势和开放问题 | 退化成论文列表 |
| method | 解释算法、数值方法或技术路线的输入、输出、步骤和边界 | 只写效果，不写机制 |
| model | 解释设备、组件或控制结构如何在 EMT 中建模 | 混淆物理对象、控制模型和仿真等效 |
| entity | 说明工具、机构、学者或组织在 EMT 生态中的角色 | 写未核查的版本、许可、功能能力 |
| test-system | 说明 benchmark 的拓扑、参数、工况和证据边界 | 把 benchmark 结果外推到所有系统 |

## 3. 证据等级

每个关键结论都必须能归入以下等级之一：

| 等级 | 含义 | 推荐写法 |
|------|------|----------|
| E1 原文事实 | 论文、手册、标准或元数据直接给出 | “作者报告……”“论文算例设置为……” |
| E2 页面内推理 | 从同一论文可合理推出 | “可理解为……”“这说明在该工况下……” |
| E3 跨论文综合 | 多篇来源共同支撑的分类或趋势 | “多篇文献中常见……”“代表性路线包括……” |
| E4 开放判断 | 编辑风险、待验证方向、边界提醒 | “需要进一步核查……”“不应外推为……” |

硬性规则：

- E1 单篇论文结果不能提升为 E3 领域共识。
- E2 推理不能写成 E1 原文事实。
- E4 开放方向不能写成已实现能力。
- 数值结果必须绑定来源、算例、指标、基准和工况；上下文不足时应删除或降级。

## 4. 强断言处理

遇到以下表达必须审查：

- “首次”“首创”“突破”“完全”“彻底”“显著”“最优”“主流”“首选”
- “高精度”“高效率”“实时”“通用”“严格保证”“无缝”
- “误差 <x%”“提升 x 倍”“适用于所有”“无需代价”
- “开源”“免费”“官方”“兼容”“原生支持”

处理规则：

1. 有明确来源和条件：保留，但绑定论文、算例、指标、版本和范围。
2. 只有单篇论文声称：改为“作者报告在该算例中……”。
3. 没有来源：改为“常见用途”“研究方向”“可通过外部流程衔接”“需要核查”。
4. 工具、许可、版本相关：没有官方资料时，写“应以官方资料为准”。
5. 方法能力绝对化：补失败边界。

示例：

- 不写：“该方法完全消除数值振荡。”
- 改写：“作者在该开关算例中报告该处理策略缓解了梯形积分数值振荡；是否适用于其他电路刚性和步长设置需要另行验证。”

## 5. 单页修订流程

1. **检查登记表**：先读 `wiki/standards/page-revision-registry.md`。`protected` 页面不得作为普通 shard 重写；`in-progress` 页面不得并发编辑。
2. **定位页面职责**：确认页面类型，不跨职责扩写。
3. **快速诊断**：标记结构缺失、证据问题和网络问题。
4. **证据采样**：taxonomy 页读取 3-8 个代表性 source 页；工具页涉及版本/许可/功能时查官方资料或降级。
5. **编辑**：短页补定义、机制、边界、代表性证据；长页先去重、修表、降级强断言。
6. **交叉引用**：只链接存在页面，并说明关系。
7. **自检**：检查无来源强断言、破表、断链、公式无解释、marker 破坏。
8. **更新登记表**：完成后将页面登记为 `protected` 或 `needs-review`，写明责任 Agent、完成日期和验证证据。

## 6. Taxonomy 推荐结构

```markdown
## 定义与边界
说明它是什么，不是什么。

## EMT 中的作用
说明它解决什么仿真问题。

## 核心机制
公式、算法、状态变量、接口变量或模型结构。

## 分类与变体
不同分支的区别和适用场景。

## 适用边界与失败模式
什么时候有效，什么时候容易失效。

## 代表性证据
用少量论文说明每类贡献，不堆长表。

## 与相关页面的关系
解释相邻方法、模型、主题的边界。

## 开放问题
只写真实未解决问题，不写宣传口号。
```

## 7. 并发工作纪律

- 开工前必须检查 `wiki/standards/page-revision-registry.md`。
- 每个 Agent 只能修改分配 shard 内页面。
- 可以读取任意页面作为证据，但不得顺手重写 shard 外页面。
- `protected` 页面默认不可修改，除非用户或协调者明确点名。
- `in-progress` 页面不可并发修改。
- 普通 shard Agent 不修改 `wiki/index.md`、`wiki/overview.md`、`wiki/log.md`、`reports/*`、审计 JSON、`.deep_enrich*.json`、`.deep_review.json`。
- 发现相邻页面定义冲突时，在当前页采用保守定义，并在交付报告中列出冲突页面和建议，不跨 shard 大范围重写。

## 8. 交付报告

每个 shard 必须报告：

- 修改页面。
- 页面类型。
- 主要修订：补了什么、删除/压缩了什么、降级了什么。
- 证据使用：读取了哪些来源，哪些结论是单篇论文结果，哪些是跨论文综合。
- 剩余风险：需要官方资料、人工确认或概念合并的事项。
- 验证命令和结果。

## 9. 验证

单个 shard 完成后至少运行：

```bash
python3 tools/audit_wiki_strict.py
python3 tools/audit_wiki_quality.py
```

批次合并或准备提交前运行：

```bash
pytest tests -q
python3 -m py_compile tools/*.py tests/test_wiki_tools.py
python3 tools/audit_wiki_strict.py
python3 tools/audit_wiki_quality.py
git diff --check
```
