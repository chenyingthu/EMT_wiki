# 仿真工具现状 (Simulation Tools Status)

## 定义与边界

仿真工具现状（Simulation Tools Status）页面用于说明如何在 EMT Wiki 中记录和比较仿真工具的能力、版本、适用场景和证据边界。它不是实时更新的产品目录，也不是工具排名。

工具能力随版本、许可、模块、硬件和用户模型库变化很快。除非引用官方资料或论文明确设置，否则页面不应写“当前支持”“官方兼容”“首选工具”“最优平台”等结论。更稳妥的写法是：某论文在某工具、某版本或某类平台上完成了某项验证；该证据支持该工具可作为该算例的仿真平台，但不自动支持该工具对所有同类问题的通用能力。

## EMT 中的作用

仿真工具信息在知识网络中承担三类角色：

- **证据上下文**：说明论文的验证基线、仿真平台、步长、模型库和后处理工具。
- **复现实验入口**：帮助读者判断是否有可能复现输入文件、脚本、模型和结果。
- **方法适用边界**：工具的求解器、元件库、实时硬件或接口限制会影响论文结论能否迁移。

因此，工具状态页面应优先记录“证据如何使用”，而不是笼统比较工具好坏。

## 核心机制

### 工具证据字段

记录工具相关信息时，至少应尝试保留：

| 字段 | 说明 |
|------|------|
| 工具名称 | PSCAD/EMTDC、EMTP-RV、ATP、RTDS、MATLAB/Simulink、CloudPSS 等 |
| 版本或模块 | 若论文或官方资料给出，应保留；没有则写未说明 |
| 仿真类型 | 离线 EMT、实时仿真、HIL、机电-电磁混合、场路耦合等 |
| 模型入口 | 图形元件库、脚本/API、MODELS/TACS、自定义代码、网表导入等 |
| 数值设置 | 步长、积分法、开关处理、初始化、求解器选项 |
| 验证基准 | 实测、解析解、其他工具、详细模型、硬件输出 |
| 证据边界 | 该工具证据只覆盖哪些算例和工况 |

缺少这些字段时，应避免把工具名称作为质量保证。

### 工具比较维度

工具比较应围绕任务需求，而不是绝对排序：

| 维度 | 应问的问题 |
|------|------------|
| 模型覆盖 | 是否有目标设备、控制器、频变模型或自定义入口 |
| 数值行为 | 是否能设置步长、插值、阻尼、初始化和事件处理 |
| 可复现性 | 是否能公开输入文件、脚本、版本和关键参数 |
| 实时性 | 是否需要硬实时、HIL 接口或固定步长硬件 |
| 互操作 | 是否需要导入网表、CIM、FMU 或外部模型 |
| 支持和许可 | 是否依赖商业模块、硬件平台或机构授权 |

这些维度应作为选择依据，而不是写成“某工具一定优于另一工具”。

## 分类与变体

| 工具类别 | 典型用途 | 证据使用方式 | 主要风险 |
|----------|----------|--------------|----------|
| 离线 EMT 工具 | 开关暂态、雷电、换流器、保护算例 | 作为详细模型或对比基线 | 默认设置和模型库差异影响波形 |
| 实时仿真器 | HIL、保护装置、控制器闭环测试 | 证明模型能在硬实时约束下运行 | 硬件资源、步长和 I/O 延迟限制 |
| 通用控制/多物理工具 | 控制器、优化、场路耦合、原型验证 | 支撑算法或接口验证 | 电力元件 EMT 细节可能不足 |
| 云/平台化工具 | 教学、协作、批量运行和 Web 工作流 | 支撑工作流和可访问性 | 版本、资源和服务状态需核查 |
| 自研求解器 | 论文算法验证、加速方法、特殊模型 | 支撑方法机制和性能测试 | 与成熟工具/实测对比不足时证据较弱 |

## 适用边界与失败模式

适用场景：

- 给 source 页补充仿真平台、基准工具和复现实验条件。
- 给 entity/tool 页补充“工具作为验证平台”的证据使用规则。
- 给方法页说明某方法是在何种工具或硬件环境下验证的。

失败模式：

- 把“论文使用某工具”写成“该工具证明该方法通用有效”。
- 把某工具某版本的模块能力写成所有版本都有。
- 用“工业标准、广泛使用、用户友好”等笼统词替代可核查证据。
- 写无来源的节点规模、误差阈值、实时步长或许可价格。
- 忽略工具默认模型、求解器设置和开关处理差异导致的波形差别。

## 修订与证据使用注意事项

- 工具名称必须和论文验证对象绑定：工具、版本、算例、步长、基准、指标。
- 涉及“当前功能、许可、模块、价格、官方支持”时，必须查官方资料；否则写“需以官方资料为准”。
- 工具比较表不应使用“优秀、强大、首选”等未定义评价词；应改为可检查维度。
- 若工具只作为输出波形对比基线，不应写成它验证了物理模型的真实性。
